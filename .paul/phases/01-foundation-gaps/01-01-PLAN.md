<objective>
## Goal
Implement standards-based email delivery using SMTP (RFC 5321) with platform-level defaults and per-tenant overrides. Tenants can configure their own SMTP server, from address, and email templates — falling back to platform defaults when not configured.

## Purpose
Email delivery is a critical gap blocking password resets, flow alerts, and notifications. The solution follows Kelta's design principles:
- **Standards-based:** SMTP (RFC 5321) via Jakarta Mail
- **Opinionated:** Works out of the box with platform SMTP defaults
- **Open source only:** No proprietary SDKs
- **Multi-tenant:** Per-tenant SMTP overrides via tenant.settings JSONB, per-tenant templates via email_template table
- **Customizable:** EmailProvider SPI for users who want non-SMTP providers

## Output
- `EmailProvider` SPI (extension point for custom providers)
- `SmtpEmailProvider` — opinionated SMTP default, supports both platform and tenant-level config
- `TenantEmailSettings` — reads per-tenant SMTP overrides from tenant.settings JSONB
- `DefaultEmailService` — orchestrates template resolution, tenant config, logging, async delivery
- JPA entities for existing `email_template` and `email_log` tables
- Internal email endpoint for cross-service calls (auth → worker)
- Password reset email delivery
- Unit tests for all new classes
</objective>

<acceptance_criteria>

## AC-1: Platform SMTP Defaults
```gherkin
Given the platform has SMTP configured via application.yml (spring.mail.*)
When a tenant has NO email overrides in their settings JSONB
Then emails are sent using the platform default SMTP server and from address
```

## AC-2: Per-Tenant SMTP Override
```gherkin
Given a tenant has email settings in their tenant.settings JSONB:
  { "email": { "smtp": { "host": "smtp.tenant.com", "port": 587, "username": "x", "password": "y" }, "fromAddress": "no-reply@tenant.com", "fromName": "Tenant Co" } }
When an email is sent for that tenant
Then the email is sent via the tenant's SMTP server with their from address (not the platform default)
```

## AC-3: Per-Tenant Email Templates
```gherkin
Given the email_template table stores templates scoped to tenant_id
When a flow or system action sends a templated email for a specific tenant
Then the tenant's version of the template is used (unique by tenant_id + name)
```

## AC-4: Provider SPI Extensibility
```gherkin
Given the EmailProvider interface defines send(EmailMessage, TenantEmailSettings)
When a user wants to add a custom provider
Then they implement EmailProvider and register it as a Spring bean — no core changes needed
```

## AC-5: Password Reset Email
```gherkin
Given a user requests a password reset via POST /auth/password/reset-request
When the reset token is generated
Then an email is sent using the tenant's SMTP config (or platform default) with a reset link
```

## AC-6: Flow Email Alert Integration
```gherkin
Given a flow contains an EMAIL_ALERT action handler
When the flow executes for a tenant with custom SMTP settings
Then the email is sent via that tenant's SMTP server
```

## AC-7: Graceful Failure
```gherkin
Given SMTP is misconfigured or unreachable (platform or tenant level)
When an email send is attempted
Then the email_log record is created with status FAILED and error_message
And the calling service does not throw an unhandled exception
```

## AC-8: Tenant SMTP Credential Security <!-- audit-added -->
```gherkin
Given a tenant stores SMTP credentials in tenant.settings JSONB
When those credentials are used, logged, or returned via API
Then the smtpPassword is NEVER included in logs, API responses, error messages, or toString() output
And credentials are only read from the database when needed for email delivery
```

## AC-9: Internal Endpoint Authorization <!-- audit-added -->
```gherkin
Given the InternalEmailController exposes POST /api/internal/email/send
When a request arrives from outside the cluster (via gateway or direct access)
Then the request is rejected with 403
And only service-to-service calls with a valid internal token are accepted
```

## AC-10: Email Log Audit Trail <!-- audit-added -->
```gherkin
Given an email is sent (success or failure)
When the email_log record is created
Then it includes: tenantId, recipientEmail, subject, source, sourceId, status, errorMessage (if failed), sentAt, and the provider used (platform or tenant SMTP host)
```

## AC-11: Idempotent Password Reset <!-- audit-added -->
```gherkin
Given a user requests a password reset multiple times within the token expiry window
When each request generates a new token
Then the previous token is invalidated (only the latest token works)
And email delivery failures do not leave stale tokens that still validate
```
