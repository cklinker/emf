<objective>
## Goal
Wire the existing scheduled_jobs table (V29) to the FlowEngine so that SCHEDULED flows execute automatically on their cron schedule. The scheduled_jobs UI already exists — this plan provides the backend executor.

## Purpose
Scheduled triggers are a stub — cron expressions are accepted and stored but never evaluated. Both Appwrite (cron functions) and Strapi (scheduled publishing) support time-based automation. This is the second critical gap in Phase 1.

## Output
- `ScheduledJobExecutorService` — polls scheduled_jobs table, executes due flows via FlowEngine
- `SchedulerConfig` — Spring TaskScheduler configuration
- `ScheduledJobRepository` — JdbcTemplate data access for scheduled_jobs
- `ScheduledJobController` — REST CRUD for scheduled jobs (if not already present)
- Cron validation on job creation
- Unit tests for all new classes
</objective>

<acceptance_criteria>

## AC-1: Cron-Based Flow Execution
```gherkin
Given a scheduled job exists with job_type=FLOW, active=true, and a valid cron expression
When the cron schedule fires (next_run_at <= now)
Then the referenced flow is executed via FlowEngine.startExecution()
And the scheduled job's last_run_at and next_run_at are updated
```

## AC-2: Timezone-Aware Scheduling
```gherkin
Given a scheduled job has timezone="America/New_York" and cron="0 9 * * *"
When the scheduler evaluates due jobs
Then the cron is evaluated in the specified timezone (not UTC)
And next_run_at is calculated in that timezone
```

## AC-3: Cron Validation
```gherkin
Given a user creates a scheduled job with an invalid cron expression
When the request is submitted
Then the API returns 400 with a descriptive error message
And the job is NOT created
```

## AC-4: Execution Logging
```gherkin
Given a scheduled flow executes (success or failure)
When the execution completes
Then the scheduled job's last_status is updated (SUCCESS or FAILED)
And the flow execution is traceable via executionId in logs
```

## AC-5: Leader Election (Single Execution)
```gherkin
Given multiple worker instances are running
When a scheduled job becomes due
Then only ONE worker instance executes the job (not duplicated)
```

## AC-6: Graceful Error Handling
```gherkin
Given a scheduled flow fails during execution
When the error is caught
Then last_status is set to FAILED
And the scheduler continues to process other due jobs
And the next_run_at is still calculated for the next cycle
```

## AC-7: Execution Audit Trail <!-- audit-added -->
```gherkin
Given a scheduled job executes (success or failure)
When the execution completes
Then a log entry records: jobId, flowId, tenantId, executionId, status, duration, and error (if failed)
And the executionId links to the flow_execution table for full step-by-step trace
```

## AC-8: Stale Job Protection <!-- audit-added -->
```gherkin
Given a scheduled job's next_run_at is more than 24 hours in the past (e.g., worker was down)
When the scheduler polls for due jobs
Then the job executes ONCE (not N times for each missed interval)
And next_run_at is recalculated from NOW, not from the stale value
```

## AC-9: Tenant Authorization on CRUD <!-- audit-added -->
```gherkin
Given a user makes a request to /api/scheduled-jobs
When the request targets a job belonging to a different tenant
Then the request is rejected (job not found for tenant)
And no cross-tenant data is leaked
```

    1. Create ScheduledJobRepository (JdbcTemplate pattern):
       - findDueJobs(): SELECT jobs WHERE active=true AND next_run_at <= NOW() ORDER BY next_run_at LIMIT 50
         - Use SELECT FOR UPDATE SKIP LOCKED for leader election (AC-5)
         - LIMIT 50 provides backpressure — prevents one poll cycle from consuming unbounded jobs
         - This ensures only one worker picks up each job in a multi-instance deployment
       - findByTenantId(tenantId): list jobs for a tenant
       - findById(id): get single job
       - create(tenantId, name, description, jobType, jobReferenceId, cronExpression, timezone, active): INSERT
       - update(id, ...): UPDATE fields
       - delete(id): DELETE
       - updateAfterExecution(id, lastStatus, lastRunAt, nextRunAt): UPDATE last_run_at, last_status, next_run_at
       - updateNextRunAt(id, nextRunAt): UPDATE next_run_at (for initial calculation on create)
       - Validate cron expression on create/update using Spring's CronExpression.parse()
       - findExecutionsByJobId(jobId, limit): list execution history for a job

    3. Create V105 migration for scheduled_job_execution table:
       ```sql
       CREATE TABLE scheduled_job_execution (
           id              VARCHAR(36)   PRIMARY KEY,
           job_id          VARCHAR(36)   NOT NULL REFERENCES scheduled_job(id) ON DELETE CASCADE,
           tenant_id       VARCHAR(36)   NOT NULL,
           flow_execution_id VARCHAR(36),
           status          VARCHAR(20)   NOT NULL,
           error_message   TEXT,
           started_at      TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
           completed_at    TIMESTAMP WITH TIME ZONE,
           duration_ms     BIGINT
       );
       CREATE INDEX idx_sje_job ON scheduled_job_execution(job_id, started_at DESC);
       ```

    4. Create SchedulerConfig (@Configuration):
       - Bean: ThreadPoolTaskScheduler (pool size=1, prefix="scheduler-")
       - @Scheduled(fixedDelayString = "${kelta.scheduler.poll-interval-ms:60000}") on a poll method
       - The poll method calls ScheduledJobExecutorService.executeAll()
       - Enabled via @ConditionalOnProperty(name = "kelta.scheduler.enabled", havingValue = "true", matchIfMissing = true)
       - Add to application.yml:
         ```yaml
         kelta:
           scheduler:
             enabled: ${SCHEDULER_ENABLED:true}
             poll-interval-ms: ${SCHEDULER_POLL_INTERVAL_MS:60000}
         ```

    Avoid: Quartz — use Spring's built-in CronExpression + @Scheduled (standards-based, no extra dependency).
    Avoid: In-memory job tracking — always read from DB (stateless worker, supports rolling restarts).

<task type="auto">
  <name>Task 2: Implement ScheduledJobExecutorService and REST controller</name>
  <files>
    kelta-worker/src/main/java/io/kelta/service/ScheduledJobExecutorService.java,

  <action>
    1. Create ScheduledJobExecutorService:
       - Inject: ScheduledJobRepository, FlowEngine, FlowStore (for loading flow definition), InitialStateBuilder, JdbcTemplate (for execution history)
       - executeAll():
         a. Query findDueJobs() (SELECT FOR UPDATE SKIP LOCKED)
         b. For each due job:
            - If job_type != FLOW: log.warn("Unsupported job type: {}", jobType), skip, recalculate next_run_at
            - Load flow definition from flow table by job_reference_id
            - If flow not found or not active: updateAfterExecution(FAILED, "Flow not found or inactive"), recalculate next_run_at, continue
            - Build initial state via InitialStateBuilder.buildFromSchedule(triggerConfig, tenantId, flowId, executionId)
            - Set TenantContext for the execution
            - Call FlowEngine.startExecution(tenantId, flowId, definitionJson, initialState, null, false)
            - Update: last_run_at=now, last_status=SUCCESS, next_run_at=calculateNext(cron, timezone)
            - Log: jobId, flowId, tenantId, executionId, status, duration <!-- audit-added -->
            - Insert execution history record into scheduled_job_execution table
         c. Catch per-job: on exception, updateAfterExecution(FAILED, error), recalculate next_run_at, insert FAILED execution history, continue to next job
         d. Clear TenantContext after each job (in finally block) <!-- audit-added -->

       - IMPORTANT: When calculating next_run_at after execution, always calculate from NOW <!-- audit-added -->
         (not from the previous next_run_at). This prevents stale jobs from firing N times
         when a worker comes back after downtime. A job that was due at 9am but executes at
         2pm should schedule its next run from 2pm, not fire 5 missed runs.

       - calculateNextRunAt(cronExpression, timezone):
         - Parse with Spring CronExpression.parse(cronExpression)
         - Calculate next from ZonedDateTime.now(ZoneId.of(timezone))
         - Return as Instant
         - If timezone is null/empty, default to UTC

    2. Create ScheduledJobController (@RestController, @RequestMapping("/api/scheduled-jobs")):
       - GET / — list jobs for tenant (from TenantContext)
       - GET /{id} — get job by id, tenant-scoped (WHERE tenant_id = ? AND id = ?) <!-- audit-added -->
       - POST / — create job (validate cron, validate timezone, calculate next_run_at, save)
       - PUT /{id} — update job (validate cron if changed, recalculate next_run_at), tenant-scoped
       - DELETE /{id} — delete job, tenant-scoped
       - POST /{id}/execute — manually trigger a job (for testing), tenant-scoped
       - POST /{id}/pause — set active=false, clear next_run_at (pause scheduling)
       - POST /{id}/resume — set active=true, recalculate next_run_at from NOW (resume at next natural interval)
       - GET /{id}/executions — list execution history for a job (from scheduled_job_execution table)
       - Cron validation: try CronExpression.parse(), on IllegalArgumentException return 400
       - Timezone validation: try ZoneId.of(timezone), on DateTimeException return 400 <!-- audit-added -->
       - ALL queries MUST filter by tenantId from TenantContext — no cross-tenant access <!-- audit-added -->

    Avoid: Running flows synchronously in the scheduler thread — FlowEngine.startExecution() is already async.
    Avoid: Holding database locks during flow execution — release lock after reading, update after execution.
    Avoid: Calculating next_run_at from previous next_run_at — always from NOW to prevent burst execution after downtime. <!-- audit-added -->
  </action>
