<objective>
## Goal
Enable programmatic API access via OAuth2 Client Credentials grant for connected apps, with a gateway filter that validates bearer tokens issued to connected apps — the same way it validates user JWT tokens.

## Purpose
API keys / programmatic access is a critical gap for CI/CD integration, scripting, and machine-to-machine communication. Both Appwrite (scoped API keys) and Strapi (3 token types) support this. Kelta already has connected_app infrastructure (V32) and OAuth2 Authorization Server (kelta-auth) — this plan wires them together.

## Design Principle
Standards-based: OAuth2 Client Credentials grant (RFC 6749 §4.4). No custom API key scheme — connected apps authenticate via `POST /oauth/token` with `grant_type=client_credentials`. The resulting access token is a standard JWT validated by the same gateway filter chain.

## Output
- Connected app OAuth2 client credentials flow end-to-end
- Token generation endpoint (already exists in kelta-auth, needs wiring)
- Token management endpoints (list, revoke) in worker
- Gateway validates connected app JWT tokens (same decoder, enriched claims)
- UI: token management on ConnectedAppsPage (generate, view, revoke)
- SDK: token generation and API key auth support
- Unit tests + e2e tests
</objective>

<acceptance_criteria>

## AC-1: Client Credentials Token Issuance
```gherkin
Given a connected app exists with active=true and is registered as an OAuth2 client
When a POST /oauth/token request is made with grant_type=client_credentials, client_id, and client_secret
Then a JWT access token is returned with tenant_id, scopes, and connected_app_id in claims
```

## AC-2: Gateway Accepts Connected App Tokens
```gherkin
Given a valid JWT access token was issued to a connected app via client credentials
When an API request is made with Authorization: Bearer <token>
Then the gateway validates the token using the same JWT decoder
And the request is authorized based on the connected app's scopes
```

## AC-3: Token Scoping
```gherkin
Given a connected app has scopes "read:contacts,write:contacts"
When the app makes a request to an endpoint not covered by its scopes
Then the request is rejected with 403 Forbidden
```

## AC-4: Token Revocation
```gherkin
Given a connected app token has been revoked via the API
When the app uses the revoked token
Then the request is rejected with 401 Unauthorized
```

## AC-5: Token Management UI
```gherkin
Given the ConnectedAppsPage exists
When a user views a connected app's details
Then they can see active tokens, generate new tokens, and revoke existing tokens
```

## AC-6: Rate Limiting Per App
```gherkin
Given a connected app has rate_limit_per_hour=1000
When the app exceeds 1000 requests in an hour
Then subsequent requests are rejected with 429 Too Many Requests
```

## AC-7: IP Restriction Enforcement
```gherkin
Given a connected app has ip_restrictions="10.0.0.0/8,192.168.1.0/24"
When a request comes from an IP outside the allowed ranges
Then the request is rejected with 403 Forbidden
```

## AC-8: Tenant Isolation <!-- audit-ready -->
```gherkin
Given connected app tokens contain tenant_id in their claims
When a connected app makes a request
Then TenantContext is set from the token's tenant_id claim
And all data access is scoped to that tenant
```

## AC-9: Audit Trail <!-- audit-ready -->
```gherkin
Given a connected app authenticates and makes requests
When requests are processed
Then the connected app's client_id is logged in request logs
And connected_app.last_used_at is updated
```

## AC-10: Token Revocation Propagation <!-- audit-added -->
```gherkin
Given a connected app token is revoked via the UI or API
When the revoked token is used within 60 seconds of revocation
Then the gateway rejects the request with 401
And the revocation check does NOT rely solely on JWT expiration
```

## AC-11: Secret Rotation Safety <!-- audit-added -->
```gherkin
Given a connected app's client secret is rotated
When the old secret is used to request a new token
Then the request is rejected with 401 (old secret invalid)
And existing tokens issued with the old secret continue to work until they expire
```

## AC-12: Token Generation Rate Limit <!-- audit-added -->
```gherkin
Given a connected app generates tokens via POST /oauth/token
When more than 10 tokens are requested in a 5-minute window
Then subsequent token requests are rejected with 429
And this prevents token generation abuse
```
