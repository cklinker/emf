<objective>
## Goal
Implement MFA/TOTP authentication with recovery codes, enabling per-tenant enforcement and per-user enrollment via admin UI and self-service setup.

## Purpose
MFA is a critical enterprise security requirement. The `platform_user.mfa_enabled` column already exists but is not enforced. This plan wires up TOTP (RFC 6238) with recovery codes, integrating into the existing login flow as a post-authentication challenge step.

## Design Principles
- **Standards-based:** TOTP (RFC 6238), 6-digit codes, 30-second period, SHA-1 (Google Authenticator compatible)
- **Opinionated defaults:** TOTP only (no SMS/push in this plan — SMS is Phase 4). 8 recovery codes, single-use.
- **Open source only:** dev.samstevens.totp library (open source, no external API calls)
- **Per-tenant configurable:** MFA can be required for all users or optional per-user

## Output
- `TotpService` — TOTP secret generation, QR code URI, code verification
- `user_totp_secret` table — encrypted TOTP secrets per user
- `user_recovery_code` table — single-use backup codes
- MFA challenge page in login flow
- MFA setup page (QR code + verification)
- Recovery code flow
- Admin MFA management API + UI
- Unit tests + e2e tests
</objective>

<acceptance_criteria>

## AC-1: TOTP Secret Generation and QR Code
```gherkin
Given a user navigates to MFA setup
When they initiate TOTP enrollment
Then a base32-encoded secret is generated
And a QR code URI is returned (otpauth://totp/Kelta:{email}?secret=...&issuer=Kelta)
And the secret is encrypted before storage (using EncryptionService)
```

## AC-2: TOTP Verification During Setup
```gherkin
Given a user has scanned the QR code in their authenticator app
When they enter the 6-digit TOTP code
Then the code is validated against the secret
And if valid, MFA is enabled for the user (mfa_enabled=true)
And 8 recovery codes are generated and returned (shown once)
```

## AC-3: MFA Challenge During Login
```gherkin
Given a user with mfa_enabled=true enters correct username/password
When they pass primary authentication
Then they are redirected to /mfa-challenge (not to the application)
And they must enter a valid TOTP code to complete login
And the session is not fully authenticated until MFA is verified
```

## AC-4: Recovery Code Authentication
```gherkin
Given a user has lost access to their authenticator app
When they click "Use recovery code" on the MFA challenge page
Then they can enter one of their 8 recovery codes
And the code is consumed (single-use) and login succeeds
And they see how many recovery codes remain
```

## AC-5: Recovery Code Regeneration
```gherkin
Given a user has used some recovery codes
When they request new recovery codes (authenticated + TOTP verification required)
Then all existing codes are invalidated
And 8 new codes are generated and shown once
```

## AC-6: MFA Disable
```gherkin
Given a user wants to disable MFA
When they provide a valid TOTP code to confirm
Then mfa_enabled is set to false
And the TOTP secret is deleted
And all recovery codes are deleted
```

## AC-7: Per-Tenant MFA Enforcement
```gherkin
Given a tenant's security policy requires MFA (mfa_required=true)
When a user without MFA logs in
Then they are redirected to MFA setup (not MFA challenge)
And they cannot access the application until MFA is enrolled
```

## AC-8: Admin MFA Management
```gherkin
Given an admin views a user's security settings
When they look at the MFA section
Then they can see MFA enrollment status
And they can reset MFA for a user (removes secret + codes, sets mfa_enabled=false)
And the reset action is logged in the security audit trail
```

## AC-9: TOTP Clock Tolerance
```gherkin
Given network latency or slight clock drift on the user's device
When they enter a TOTP code
Then codes from the previous and next 30-second windows are also accepted (±1 window)
```

## AC-10: Admin UI for MFA Setup
```gherkin
Given a user navigates to their security settings
When they click "Enable MFA"
Then they see a QR code to scan
And they can enter the verification code
And upon success they see their recovery codes (download option)
And after dismissing, the codes are never shown again
```

## AC-11: MFA Challenge Rate Limiting <!-- audit-added -->
```gherkin
Given an attacker has stolen a user's password
When they attempt to brute-force the MFA challenge
Then after 5 consecutive failed TOTP/recovery code attempts, the MFA challenge is locked for 5 minutes
And failed MFA attempts are logged with client IP for security monitoring
And the lockout is independent of the password lockout (separate counter)
```

## AC-12: TOTP Code Replay Prevention <!-- audit-added -->
```gherkin
Given a valid TOTP code is used to authenticate
When the same code is submitted again within the same 30-second window
Then the second submission is rejected
And the last-used TOTP timestamp is tracked per user to prevent replay
```

## AC-13: MFA Event Audit Logging <!-- audit-added -->
```gherkin
Given any MFA-related action occurs
When the action completes (success or failure)
Then a structured security audit log entry is created with:
  event type (MFA_CHALLENGE_SUCCESS, MFA_CHALLENGE_FAILED, MFA_ENROLLED, MFA_DISABLED, MFA_RESET, RECOVERY_CODE_USED)
  actor (userId), target (userId), result, client IP, timestamp
```

## AC-14: Re-authentication Before MFA Changes <!-- audit-added -->
```gherkin
Given a user wants to enable, disable, or regenerate recovery codes for MFA
When they initiate the action via self-service
Then they must re-enter their password before the operation proceeds
And this prevents session hijacking from modifying MFA settings
```

## AC-15: Tenant-Scoped User Lookup for MFA <!-- audit-added -->
```gherkin
Given multiple users across tenants share the same email
When MFA state is loaded during login
Then the MFA check uses the correct tenant-scoped user (not first-match)
And this is consistent with the tenant context resolved from the login URL
```

    2. Create V107 migration:
       ```sql
       CREATE TABLE user_totp_secret (
           id          VARCHAR(36) PRIMARY KEY DEFAULT gen_random_uuid()::text,
           user_id     VARCHAR(36) NOT NULL UNIQUE REFERENCES platform_user(id),
           secret      VARCHAR(500) NOT NULL,  -- AES-256-GCM encrypted base32 secret
           verified    BOOLEAN NOT NULL DEFAULT false,
           last_used_at BIGINT,                -- epoch second of last accepted TOTP code (replay prevention) <!-- audit-added -->
           mfa_failed_attempts INTEGER NOT NULL DEFAULT 0,  -- rate limiting counter <!-- audit-added -->
           mfa_locked_until TIMESTAMP WITH TIME ZONE,       -- MFA-specific lockout <!-- audit-added -->
           created_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW(),
           updated_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
       );

       CREATE TABLE user_recovery_code (
           id          VARCHAR(36) PRIMARY KEY DEFAULT gen_random_uuid()::text,
           user_id     VARCHAR(36) NOT NULL REFERENCES platform_user(id),
           code_hash   VARCHAR(500) NOT NULL,  -- BCrypt hash
           used        BOOLEAN NOT NULL DEFAULT false,
           created_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
       );
       CREATE INDEX idx_recovery_code_user ON user_recovery_code(user_id);

       -- Add mfa_required to password_policy for per-tenant enforcement
       ALTER TABLE password_policy ADD COLUMN mfa_required BOOLEAN NOT NULL DEFAULT false;
       ```

    3. Create TotpService:
       - Inject: JdbcTemplate, PasswordEncoder, EncryptionService
       - generateSecret(): create base32 secret using dev.samstevens.totp.secret.DefaultSecretGenerator
       - getQrCodeUri(email, secret): return otpauth URI (issuer=Kelta, period=30, digits=6, algorithm=SHA1)
       - verifyCode(secret, code): validate TOTP with ±1 window tolerance using dev.samstevens.totp.code.DefaultCodeVerifier
       - enrollUser(userId, secret, code): verify code, then encrypt and store secret, set mfa_enabled=true, generate recovery codes
       - generateRecoveryCodes(userId): generate 8 random codes (format: xxxx-xxxx using SecureRandom, alphanumeric charset [a-z0-9], 8 chars = ~41 bits entropy per code), BCrypt hash each, store, return plaintext list <!-- audit-added: entropy spec -->
       - verifyRecoveryCode(userId, code): check unhashed code against stored hashes, mark as used, return success/failure
       - trackLastUsedCode(userId, timestamp): store last successful TOTP timestamp to prevent replay within same window <!-- audit-added -->
       - checkMfaRateLimit(userId): track failed MFA attempts, lock after 5 consecutive failures for 5 minutes <!-- audit-added -->
       - disableMfa(userId, code): verify TOTP code, delete secret + recovery codes, set mfa_enabled=false
       - resetMfa(userId): admin action — delete secret + codes, set mfa_enabled=false (no TOTP verification)
       - loadSecret(userId): load encrypted secret, decrypt with EncryptionService
       - isEnrolled(userId): check if user_totp_secret exists AND verified=true

    4. TotpServiceTest:
       - generateSecret: returns non-null base32 string
       - getQrCodeUri: contains otpauth://totp/Kelta: prefix with email and secret
       - verifyCode: valid code returns true, invalid returns false
       - verifyCode replay: same code rejected within same 30-second window <!-- audit-added -->
       - enrollUser: stores encrypted secret, sets mfa_enabled, generates 8 recovery codes
       - verifyRecoveryCode: valid code returns true and marks used, reuse fails
       - generateRecoveryCodes: returns 8 codes in xxxx-xxxx format with SecureRandom <!-- audit-added -->
       - checkMfaRateLimit: locks after 5 failures, resets on success <!-- audit-added -->
       - disableMfa: deletes secret and codes, sets mfa_enabled=false
       - resetMfa: works without TOTP verification (admin action)

    Avoid: Storing plaintext TOTP secrets — always encrypt via EncryptionService.
    Avoid: SMS/push MFA — TOTP only in this plan (SMS is Phase 4).
