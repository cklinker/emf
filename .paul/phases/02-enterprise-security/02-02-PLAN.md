<objective>
## Goal
Add write-side field-level security enforcement to complement the existing read-side stripping. When a user PUTs/PATCHes a record, HIDDEN and READ_ONLY fields must be silently stripped from the request body before persistence.

## Purpose
Read-side field security already works via `CerbosFieldSecurityAdvice` — hidden fields are stripped from responses. But the write side is unprotected: users can submit writes to HIDDEN/READ_ONLY fields, and those writes persist. This creates a data integrity risk where users can modify fields they shouldn't see, and a compliance gap where field-level security is one-directional.

## Output
- `CerbosFieldWriteSecurityAdvice` — request-side interceptor that strips denied fields from PUT/PATCH bodies
- Unit tests for write-side enforcement
- Additional unit tests for the existing read-side advice
</objective>

## Key Architecture Facts
- CerbosFieldSecurityAdvice (read-side): ResponseBodyAdvice @Order(10), strips fields from response
- CerbosRecordAuthorizationAdvice (record-level): ResponseBodyAdvice @Order(0)
- CerbosAuthorizationService: batchCheckFieldAccess(email, profileId, tenantId, collectionId, fieldIds, action)
  - action can be "read" or "write"
- Field permissions stored in field_permission table: VISIBLE, READ_ONLY, HIDDEN
- Cerbos policies generated by CerbosPolicyGenerator from field_permission data
- System fields (createdAt, updatedAt, createdBy, updatedBy) bypass security checks
- Admin/metadata paths bypass field security
- CerbosPermissionResolver extracts identity from request headers (X-User-Email, X-User-Profile-Id, etc.)

