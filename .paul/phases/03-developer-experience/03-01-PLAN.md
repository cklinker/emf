<objective>
## Goal
Implement JSON:API Atomic Operations extension (https://jsonapi.org/ext/atomic) for bulk CRUD — allowing clients to create, update, and delete multiple records across collections in a single atomic request with transactional guarantees.

## Purpose
Batch operations are critical for developer experience: importing data, bulk updates, and multi-record workflows. The JSON:API Atomic Operations extension is the standard way to do this. The research (`.paul/research/jsonapi-atomic-operations.md`) already defines the approach.

## Design Principles
- **Standards-based:** JSON:API Atomic Operations extension (exact spec compliance)
- **Opinionated defaults:** Max 100 operations per request (configurable per-tenant, hard ceiling 500)
- **Transactional:** All operations in a single database transaction — all succeed or all rollback
- **Local ID support:** `lid` (local identifier) for referencing server-generated IDs within same request

## Output
- `POST /api/operations` endpoint with JSON:API Atomic Operations support
- `AtomicOperationExecutor` — orchestrates operations within a transaction
- JSON:API extension types: AtomicOperation, AtomicResult
- Per-tenant configurable operation limits
- Unit tests
</objective>

<acceptance_criteria>

## AC-1: Atomic Operations Endpoint
```gherkin
Given an authenticated user
When they POST to /api/operations with Content-Type: application/vnd.api+json
Then the request body is parsed as an array of atomic:operations
And each operation is executed sequentially within a single transaction
And the response contains atomic:results matching operation order
```

## AC-2: Add (Create) Operations
```gherkin
Given an atomic operation with "op": "add"
When the operation includes type, attributes, and optional lid
Then a new record is created in the specified collection
And the result contains the created resource with server-generated id
And if lid was provided, the lid-to-id mapping is tracked for later operations
```

## AC-3: Update Operations
```gherkin
Given an atomic operation with "op": "update"
When the operation includes ref (type + id) and data with attributes
Then the specified record is updated
And the result contains the updated resource
```

## AC-4: Remove Operations
```gherkin
Given an atomic operation with "op": "remove"
When the operation includes ref (type + id)
Then the specified record is deleted
And the result is an empty object
```

## AC-5: Local ID (lid) Resolution
```gherkin
Given an "add" operation creates a record with lid "temp-1"
When a subsequent operation references lid "temp-1" in a relationship or ref
Then the lid is resolved to the server-generated id
And the subsequent operation uses the real id
```

## AC-6: Transactional Atomicity
```gherkin
Given a batch of 5 operations where operation 3 fails validation
When the batch is executed
Then operations 1 and 2 are rolled back
And operations 4 and 5 are never executed
And the response is a 422 error with the failing operation index
```

## AC-7: Operation Limit Enforcement
```gherkin
Given a tenant has a max_batch_operations limit of 100 (default)
When a request contains more than 100 operations
Then the request is rejected with 422 and error message "Too many operations"
And no operations are executed
```

## AC-8: Per-Operation Authorization
```gherkin
Given a user with create permission on "contacts" but not "accounts"
When a batch includes an "add" for contacts and an "add" for accounts
Then the entire batch fails with 403
And no operations are committed (authorization checked before execution)
```

## AC-9: Error Response Format
```gherkin
Given an operation fails
When the error response is constructed
Then it follows JSON:API error format with source.pointer indicating the failing operation index
And the response includes the original operation for debugging
```

## AC-10: Rate Limiting Counts Batch Operations <!-- audit-added -->
```gherkin
Given a batch request with N operations
When the gateway rate limiter evaluates the request
Then the request counts as N API calls against the tenant's rate limit (not 1)
And a batch that would exceed the rate limit is rejected with 429
```

## AC-11: Kafka Events Fire for Each Operation <!-- audit-added -->
```gherkin
Given a batch of create/update/delete operations
When the batch executes through QueryEngine
Then RecordEventPublisher fires a Kafka event for each individual operation
And all events are published after the transaction commits (not during)
```

## AC-12: Idempotency Key Support <!-- audit-added -->
```gherkin
Given a client sends an Idempotency-Key header with a batch request
When the same Idempotency-Key is used for a retry
Then the server returns the cached result from the first execution
And no operations are re-executed
```

  <action>
    1. Create AtomicOperation record in runtime-jsonapi:
       - op: String ("add", "update", "remove")
       - ref: ResourceIdentifier (type + id, for update/remove)
       - data: ResourceObject (type, id/lid, attributes, relationships)
       - Validation: op is required, add needs data, update/remove need ref

    2. Create AtomicResult record:
       - data: ResourceObject (result of add/update, null for remove)
       - Empty result for remove operations

    3. Create AtomicOperationExecutor in runtime-core:
       - Inject: QueryEngine, CollectionRegistry (to resolve type → CollectionDefinition)
       - execute(List<AtomicOperation>, tenantId): returns List<AtomicResult>
       - MUST be called within a @Transactional context (caller provides transaction)
       - Maintains Map<String, String> lidMap (lid → server-generated id)
       - For each operation:
         a. Resolve collection definition from type
         b. If "add": extract attributes from data, create via QueryEngine, track lid→id
         c. If "update": resolve ref (replace lid if needed), update via QueryEngine
         d. If "remove": resolve ref (replace lid if needed), delete via QueryEngine
         e. Record result
       - On any failure: throw exception (caller's @Transactional will rollback)

    4. AtomicOperationExecutorTest:
       - Execute add operation → creates record, returns result with id
       - Execute update operation → updates record, returns result
       - Execute remove operation → deletes record, returns empty result
       - lid resolution: add with lid, then update referencing lid → uses real id
       - Failure on operation N → exception thrown (transaction rollback tested at controller level)
       - Invalid operation type → exception
       - Missing ref on update/remove → exception

    Avoid: Putting transaction management in the executor — that's the controller's job.
    Avoid: Parallel execution — operations must be sequential (lid resolution depends on order).
  </action>
