<objective>
## Goal
Add on-the-fly image transformation capabilities: resize, crop, and format conversion via URL query parameters on the file serving endpoint.

## Purpose
Image transformations are essential for responsive UIs — serving appropriately sized thumbnails, avatars, and previews without storing multiple versions. Transformations happen on-the-fly at request time, with optional caching.

## Design Principles
- **Open source only:** Thumbnailator library (pure Java, no native dependencies)
- **URL-driven:** Transform parameters in query string (e.g., `?w=200&h=200&fit=cover&format=webp`)
- **Built on existing infrastructure:** Extends FileController and S3StorageService from plan 02-03

## Output
- `ImageTransformService` — resize, crop, format conversion
- Extended `ImageController` — transform parameters on file serving
- Unit tests
</objective>

<acceptance_criteria>

## AC-1: Resize by Width
```gherkin
Given an image file in S3
When requested with ?w=200
Then the image is resized to 200px wide with proportional height
And the response Content-Type matches the output format
```

## AC-2: Resize by Height
```gherkin
Given an image file in S3
When requested with ?h=150
Then the image is resized to 150px tall with proportional width
```

## AC-3: Resize with Fit Mode
```gherkin
Given an image file in S3
When requested with ?w=200&h=200&fit=cover
Then the image is resized and cropped to exactly 200x200
And when requested with ?w=200&h=200&fit=contain
Then the image fits within 200x200 maintaining aspect ratio
```

## AC-4: Format Conversion
```gherkin
Given a PNG image in S3
When requested with ?format=webp
Then the response is a WebP image
And Content-Type is image/webp
And when requested with ?format=jpeg&quality=80
Then the response is a JPEG with 80% quality
```

## AC-5: No Transform Passthrough
```gherkin
Given an image file requested without any transform parameters
When served via the file endpoint
Then the original file is streamed without modification
And no image processing overhead is incurred
```

## AC-6: Non-Image Passthrough
```gherkin
Given a PDF or other non-image file
When requested with transform parameters
Then the transform parameters are ignored
And the original file is served unchanged
```

## AC-7: Max Dimensions Enforcement
```gherkin
Given a transform request with w=10000
When the max dimension limit is 4096 (default)
Then the dimension is clamped to 4096
And no excessively large images are generated
```

## AC-8: Image Bomb Protection <!-- audit-added -->
```gherkin
Given a compressed image that decompresses to > 20 megapixels
When transformation is attempted
Then the request is rejected with 422 before full decompression
And a warning is logged
```

## AC-9: Transform Concurrency Limit <!-- audit-added -->
```gherkin
Given the worker is processing image transforms
When more than 4 concurrent transforms are in progress
Then additional requests wait (bounded semaphore)
And the worker heap is protected from exhaustion
```

       - Max dimension: clamp width/height to 4096
       - Supported input formats: JPEG, PNG, GIF, BMP, TIFF

       ```java
       public record TransformResult(byte[] data, String contentType) {}
       ```

       - Contain fit preserves aspect ratio within bounds
       - Format conversion changes content type
       - Max dimension clamping
       - Null params returns null (passthrough signal)

    Avoid: Loading very large images — check source dimensions first via ImageReader metadata, reject > 20MP BEFORE decompression. <!-- audit-added -->
    Avoid: WebP dependency issues — fall back to PNG if WebP not supported by JVM.

    5. Add concurrency guard: use Semaphore(4) to limit concurrent transforms. <!-- audit-added -->
       Prevents heap exhaustion from parallel large-image processing.
       Acquire before transform, release in finally block.
