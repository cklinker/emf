<objective>
## Goal
Generate an OpenAPI 3.0 specification dynamically from collection schemas, serving it as both JSON and a Swagger UI page for interactive API exploration.

## Purpose
Auto-generated API documentation eliminates the burden of maintaining separate docs, ensures accuracy as collections evolve, and provides developers with interactive endpoints for testing. The documentation reflects the actual tenant's collections and fields.

## Design Principles
- **Standards-based:** OpenAPI 3.0 specification (industry standard)
- **Dynamic:** Generated from live CollectionRegistry — always reflects current schema
- **Zero maintenance:** No manual documentation files to keep in sync

## Output
- `OpenApiGenerator` — builds OpenAPI 3.0 spec from CollectionDefinition records
- `OpenApiController` — serves spec as JSON and Swagger UI HTML
- Unit tests
</objective>

## Key Architecture Facts
- CollectionDefinition: name, displayName, description, fields, readOnly, systemCollection
- FieldDefinition: name, type (FieldType enum), nullable, immutable, unique, defaultValue, validationRules
- FieldType: STRING, INTEGER, LONG, DOUBLE, BOOLEAN, DATE, DATETIME, JSON, etc.
- CollectionRegistry: get(name), getAll() — returns all registered collections
- DynamicCollectionRouter: GET, POST, PUT, PATCH, DELETE on /api/{collection}
- JSON:API response format with data, attributes, relationships, included

## AC-1: OpenAPI JSON Endpoint
```gherkin
Given collections are registered in the system
When a user requests GET /api/docs/openapi.json
Then a valid OpenAPI 3.0 JSON specification is returned
And it includes paths for each collection's CRUD endpoints
And it includes schemas for each collection's fields
```

## AC-2: Swagger UI Endpoint
```gherkin
Given the OpenAPI spec is available
When a user navigates to /api/docs
Then an interactive Swagger UI page is rendered
And the UI loads the OpenAPI spec from /api/docs/openapi.json
```

## AC-3: Collection-Specific Paths
```gherkin
Given a collection "contacts" with fields name (STRING), email (STRING), age (INTEGER)
When the OpenAPI spec is generated
Then it includes:
  GET /api/contacts — list with pagination params
  GET /api/contacts/{id} — get by ID
  POST /api/contacts — create with request body schema
  PUT /api/contacts/{id} — full update
  PATCH /api/contacts/{id} — partial update
  DELETE /api/contacts/{id} — delete
```

## AC-4: Field Type Mapping to OpenAPI Types
```gherkin
Given FieldType values in the system
When mapped to OpenAPI types
Then STRING → type: string
And INTEGER → type: integer
And DOUBLE → type: number, format: double
And BOOLEAN → type: boolean
And DATE → type: string, format: date
And DATETIME → type: string, format: date-time
And JSON → type: object
```

## AC-5: Read-Only Collections
```gherkin
Given a collection with readOnly=true
When the OpenAPI spec is generated
Then only GET endpoints are included (no POST, PUT, PATCH, DELETE)
```

## AC-6: Authentication Documentation
```gherkin
Given the OpenAPI spec
When security schemes are reviewed
Then it documents Bearer JWT authentication
And all collection endpoints require the security scheme
```

## AC-7: Docs Require Authentication <!-- audit-added -->
```gherkin
Given the docs endpoints (/api/docs, /api/docs/openapi.json)
When accessed without authentication headers
Then a 401 is returned
And the tenant's schema is not exposed to unauthenticated users
```

## AC-8: Atomic Operations Documented <!-- audit-added -->
```gherkin
Given the OpenAPI spec
When the paths section is reviewed
Then POST /api/operations is documented
With request body schema for atomic:operations array
And response schema for atomic:results
```
