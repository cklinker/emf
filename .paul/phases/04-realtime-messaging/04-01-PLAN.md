<objective>
## Goal
Add WebSocket realtime subscriptions to the gateway, allowing clients to subscribe to record change events for specific collections and receive live updates via Kafka-bridged WebSocket connections.

## Purpose
Realtime updates are essential for collaborative and responsive UIs. Instead of polling, clients subscribe to collection changes and receive push notifications when records are created, updated, or deleted. The gateway bridges Kafka record events to WebSocket connections, filtered by tenant and collection.

## Design Principles
- **Standards-based:** WebSocket (RFC 6455) with JSON message protocol
- **Permission-aware:** Only deliver events for collections the user can read (Cerbos check)
- **Kafka-bridged:** Gateway consumes `kelta.record.changed` topic and fans out to WebSocket sessions
- **Tenant-scoped:** Subscriptions are isolated by tenant (users only see their tenant's events)

## Output
- WebSocket endpoint: `ws://gateway/ws/realtime`
- `RealtimeWebSocketHandler` — handles connections, subscriptions, JWT auth
- `SubscriptionManager` — tracks active subscriptions per session
- `RealtimeKafkaBridge` — Kafka consumer that routes events to WebSocket sessions
- Unit tests
</objective>

## Key Architecture Facts
- Gateway is Spring Cloud Gateway (WebFlux/reactive) — native WebSocket support
- Kafka topic kelta.record.changed: partitioned by tenantId:collectionName
- PlatformEvent<RecordChangedPayload>: eventType, tenantId, userId, collectionName, recordId, changeType, data
- ChangeType enum: CREATED, UPDATED, DELETED
- JWT auth via DynamicReactiveJwtDecoder — can authenticate WebSocket handshake
- SecurityConfig is reactive (@EnableWebFluxSecurity)
- No WebSocket dependencies currently in gateway

<acceptance_criteria>

## AC-1: WebSocket Connection with JWT Auth
```gherkin
Given a client connects to ws://gateway/ws/realtime
When the connection handshake includes a valid JWT token (via query param or protocol header)
Then the connection is established and the tenant context is resolved
And if no valid JWT: connection rejected with 401
```

## AC-2: Subscribe to Collection Events
```gherkin
Given an authenticated WebSocket connection
When the client sends: {"action":"subscribe","collection":"contacts"}
Then the subscription is registered
And the client receives: {"action":"subscribed","collection":"contacts"}
And future record events for "contacts" in the user's tenant are delivered
```

## AC-3: Unsubscribe from Collection
```gherkin
Given an active subscription
When the client sends: {"action":"unsubscribe","collection":"contacts"}
Then the subscription is removed
And no further events for "contacts" are delivered
```

## AC-4: Event Delivery
```gherkin
Given a client subscribed to "contacts"
When a record in "contacts" is created/updated/deleted (Kafka event)
Then the client receives: {"event":"record.changed","collection":"contacts","changeType":"CREATED","recordId":"...","data":{...}}
And the event is only delivered to sessions subscribed to that collection+tenant
```

## AC-5: Tenant Isolation
```gherkin
Given two users from different tenants both subscribed to "contacts"
When a record event occurs for tenant-A
Then only the tenant-A user receives the event
And the tenant-B user receives nothing
```

## AC-6: Permission-Aware Delivery
```gherkin
Given a user without read permission on "contacts" (Cerbos denies)
When they attempt to subscribe to "contacts"
Then the subscription is rejected with: {"action":"error","message":"Permission denied"}
And no events are delivered for that collection
```

## AC-7: Connection Cleanup
```gherkin
Given an active WebSocket connection with subscriptions
When the connection is closed (client disconnect, network failure)
Then all subscriptions for that session are cleaned up
And no orphaned subscriptions remain
```

## AC-8: Heartbeat / Keep-Alive
```gherkin
Given an idle WebSocket connection
When no messages are exchanged for 30 seconds
Then the server sends a ping frame
And if no pong is received within 10 seconds, the connection is closed
```

## AC-9: Subscription Limit Per Session <!-- audit-added -->
```gherkin
Given an authenticated WebSocket connection
When the client subscribes to more than 50 collections
Then the subscription is rejected with: {"action":"error","message":"Subscription limit reached"}
And the connection remains open with existing subscriptions
```

## AC-10: Field-Level Security on Events <!-- audit-added -->
```gherkin
Given a user with HIDDEN fields on "contacts"
When a record.changed event for "contacts" is delivered via WebSocket
Then HIDDEN fields are stripped from the event data before delivery
And this is consistent with CerbosFieldSecurityAdvice read-side behavior
```

## AC-11: Connection Limit Per Tenant <!-- audit-added -->
```gherkin
Given a tenant has reached the max WebSocket connections (default 100)
When a new connection is attempted
Then the connection is rejected with close code 1008 (Policy Violation)
And a warning is logged
```

## AC-12: JWT Expiration Handling <!-- audit-added -->
```gherkin
Given a long-lived WebSocket connection
When the JWT used at handshake expires
Then the connection is closed with close code 4001 (Token Expired)
And the client must reconnect with a fresh token
```

  <action>
    1. Add WebSocket dependency to gateway pom.xml:
       Spring WebFlux already includes WebSocket support — no additional dependency needed.
       (spring-cloud-starter-gateway includes spring-boot-starter-webflux)

    2. Create WebSocketConfig:
       - Register RealtimeWebSocketHandler at path /ws/realtime
       - Configure WebSocket handshake with JWT extraction from query param (?token=xxx)
       - Set ping/pong interval (30s ping, 10s timeout)

    3. Create SubscriptionManager:
       - Thread-safe registry: Map<sessionId, Set<String>> (sessionId → subscribed collections)
       - Reverse index: Map<tenantId:collectionName, Set<WebSocketSession>> for event routing
       - subscribe(session, tenantId, collectionName): add to both maps
       - unsubscribe(session, collectionName): remove from both maps
       - removeSession(session): clean up all subscriptions for session
       - getSubscribers(tenantId, collectionName): returns Set<WebSocketSession>
       - Use ConcurrentHashMap + ConcurrentHashMap<String, CopyOnWriteArraySet>

    4. Create RealtimeWebSocketHandler (implements WebSocketHandler):
       - handle(WebSocketSession): main connection lifecycle
       - On connect: extract JWT from query param, validate via DynamicReactiveJwtDecoder,
         extract tenantId from JWT claims, store in session attributes
       - On message (JSON): parse action (subscribe/unsubscribe)
         - subscribe: check Cerbos read permission, check subscription limit (max 50), then register <!-- audit-added -->
         - unsubscribe: remove from SubscriptionManager
       - On close: call SubscriptionManager.removeSession()
       - Send messages via session.send(Mono.just(session.textMessage(json)))
       - Track JWT expiration time — schedule connection close when token expires <!-- audit-added -->
       - Track connection count per tenant — reject if >100 per tenant <!-- audit-added -->

    5. SubscriptionManagerTest:
       - Subscribe adds to both forward and reverse index
       - Unsubscribe removes correctly
       - removeSession cleans all subscriptions
       - getSubscribers returns only matching sessions
       - Thread safety (concurrent subscribe/unsubscribe)

    6. RealtimeWebSocketHandlerTest:
       - Reject connection without JWT
       - Accept connection with valid JWT
       - Subscribe message registers subscription
       - Unsubscribe message removes subscription

    Avoid: Using STOMP — raw WebSocket with JSON is simpler and more flexible.
    Avoid: Storing subscriptions in Redis — in-memory is fine since subscriptions are per-gateway-instance.
  </action>

         }

       - Log event routing metrics (events routed, sessions notified)

    2. Consumer group strategy:
       - Use a SEPARATE consumer group from the existing gateway consumer (e.g., "kelta-gateway-realtime")
       - This ensures realtime events are consumed independently from config event processing

    Avoid: Blocking the Kafka consumer thread on WebSocket sends — use async/reactive sends.
