<objective>
## Goal
Add SMS-based authentication: OTP verification codes for passwordless login and as an additional MFA factor. Uses an SPI pattern with a log-only default provider (open source only in core) and extension points for Twilio/Vonage/etc.

## Purpose
SMS authentication enables passwordless login flows and provides an alternative MFA channel to TOTP for users who prefer phone-based verification. Following the project's open source principle, the core ships with a log-only provider for development/testing, while production deployments plug in their preferred SMS gateway via the SPI.

## Design Principles
- **Open source only:** Log-only default provider (no Twilio in core)
- **SPI extensible:** `SmsProvider` interface mirrors `EmailProvider` pattern
- **Standards-based:** OTP codes (6 digits, 5-minute expiry, rate limited)
- **Secure:** Codes are hashed before storage, rate limited per phone number

## Output
- `SmsProvider` SPI interface + `LogOnlySmsProvider` default
- `SmsVerificationService` — OTP generation, storage, verification
- `SmsAuthController` — SMS login/verify endpoints
- `sms_verification` table for OTP storage
- Unit tests
</objective>

<acceptance_criteria>

## AC-1: SmsProvider SPI
```gherkin
Given the platform's open source principle
When SMS functionality is needed
Then an SmsProvider interface exists with send(SmsMessage) method
And a LogOnlySmsProvider logs the message (development/testing default)
And custom providers (Twilio, etc.) can be registered as Spring beans
```

## AC-2: OTP Code Generation and Storage
```gherkin
Given a user requests SMS verification
When an OTP code is generated
Then a 6-digit numeric code is created using SecureRandom
And the code is hashed (SHA-256) before storage in sms_verification table
And the code expires after 5 minutes
And only the most recent code per phone number is valid
```

## AC-3: SMS Verification Endpoint
```gherkin
Given a user provides their phone number
When they request POST /auth/sms/send
Then an OTP code is generated and sent via SmsProvider
And the response confirms "Code sent" (does not reveal code)
And the user can verify via POST /auth/sms/verify with the code
```

## AC-4: Rate Limiting on SMS Send
```gherkin
Given a phone number
When more than 3 OTP codes are requested within 5 minutes
Then subsequent requests are rejected with 429
And no SMS is sent
```

## AC-5: SMS as MFA Factor
```gherkin
Given a user has a phone number on their profile
When they configure SMS as their MFA method (alternative to TOTP)
Then the MFA challenge page offers SMS as an option
And verifying the SMS code completes the MFA challenge
```

## AC-6: Phone Number Validation
```gherkin
Given a phone number input
When it is validated
Then it must match E.164 format (+{country}{number})
And invalid formats are rejected with a clear error message
```

## AC-7: Constant-Time Code Comparison <!-- audit-added -->
```gherkin
Given a verification code is submitted
When compared against the stored hash
Then the comparison uses MessageDigest.isEqual() (constant-time)
And timing attacks cannot determine partial matches
```

## AC-8: SMS Event Audit Logging <!-- audit-added -->
```gherkin
Given an SMS send or verify event occurs
When the operation completes
Then a security audit log entry is emitted with: event type, phone (masked), tenant, result
And phone numbers are partially masked in logs (+1***567890)
```

## AC-9: Expired Code Cleanup <!-- audit-added -->
```gherkin
Given expired sms_verification rows exist
When a new code is generated for the same phone
Then expired/used rows for that phone are deleted
And the table does not grow unbounded
```

    1. Create SmsProvider interface (mirrors EmailProvider):
       ```java
       public interface SmsProvider {
           void send(SmsMessage message) throws SmsDeliveryException;
       }
       ```

    2. Create SmsMessage record:
       ```java
       public record SmsMessage(String to, String body) {}
       ```

    3. Create LogOnlySmsProvider (default):
       - @ConditionalOnMissingBean(SmsProvider.class)
       - Logs the message at INFO level (for development/testing)
       - Log format: "SMS [to=+1234567890]: Your code is 123456"

    4. Create DefaultSmsService:
       - Inject: SmsProvider, JdbcTemplate
       - sendVerificationCode(phoneNumber, tenantId):
         a. Validate E.164 format
         b. Check rate limit (3 per 5 min per phone)
         c. Generate 6-digit code via SecureRandom
         d. Hash code (SHA-256) and store in sms_verification with 5-min expiry
         e. Send via SmsProvider: "Your Kelta verification code is: {code}"
         f. Return success/failure
       - verifyCode(phoneNumber, code, tenantId):
         a. Load most recent unexpired verification for phone
         b. SHA-256 hash the provided code and compare using MessageDigest.isEqual() (constant-time) <!-- audit-added -->
         c. If match: mark as used, return success
         d. If no match: increment attempts, return failure
         e. Max 5 verification attempts per code
       - Before inserting new code: delete expired/used rows for the same phone (cleanup) <!-- audit-added -->
       - Log all SMS events via security.audit logger (mask phone: +1***567890) <!-- audit-added -->

    5. Create V108 migration:
       ```sql
       CREATE TABLE sms_verification (
           id          VARCHAR(36) PRIMARY KEY DEFAULT gen_random_uuid()::text,
           phone       VARCHAR(20) NOT NULL,
           code_hash   VARCHAR(128) NOT NULL,
           tenant_id   VARCHAR(36) NOT NULL,
           expires_at  TIMESTAMP WITH TIME ZONE NOT NULL,
           used        BOOLEAN NOT NULL DEFAULT false,
           attempts    INTEGER NOT NULL DEFAULT 0,
           created_at  TIMESTAMP WITH TIME ZONE NOT NULL DEFAULT NOW()
       );
       CREATE INDEX idx_sms_verification_phone ON sms_verification(phone, tenant_id, created_at DESC);

       -- Add phone_number to platform_user
       ALTER TABLE platform_user ADD COLUMN phone_number VARCHAR(20);
       ```

    6. DefaultSmsServiceTest:
       - sendVerificationCode generates 6-digit code
       - sendVerificationCode validates E.164 format
       - sendVerificationCode rate limits at 3 per 5 min
       - verifyCode matches correct code
       - verifyCode rejects wrong code
       - verifyCode rejects expired code
       - verifyCode limits attempts to 5

    Avoid: Including Twilio SDK — log-only default is the open source path.
    Avoid: Storing plaintext OTP codes — always SHA-256 hash.
