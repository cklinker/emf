<objective>
## Goal
Add push notification infrastructure: device registration, notification delivery via SPI, and an admin API for sending targeted notifications. Ships with a log-only default provider; production deployments plug in FCM/APNs via the SPI.

## Purpose
Push notifications enable proactive user engagement — alerting users to record changes, workflow completions, and system events on their mobile devices. Following the established SPI pattern (EmailProvider, SmsProvider), the core is open source with extension points for proprietary push services.

## Design Principles
- **Open source only:** Log-only default provider (no Firebase SDK in core)
- **SPI extensible:** `PushProvider` interface for FCM, APNs, or any push service
- **Device registration:** Users register device tokens, scoped by tenant + user
- **Targeted delivery:** Send to specific users, tenant-wide, or topic-based

## Output
- `PushProvider` SPI + `LogOnlyPushProvider` default
- `DefaultPushService` — device management + notification routing
- `PushDeviceController` — device registration + admin notification API
- `push_device` table for device token storage
- Unit tests
</objective>

<acceptance_criteria>

## AC-1: PushProvider SPI
```gherkin
Given the platform's open source principle
When push notification functionality is needed
Then a PushProvider interface exists with send(PushMessage) method
And a LogOnlyPushProvider logs the notification (development/testing default)
And custom providers (FCM, APNs) can be registered as Spring beans
```

## AC-2: Device Registration
```gherkin
Given an authenticated user with a mobile device
When they register a device token via POST /api/devices
Then the token is stored with: userId, tenantId, platform (ios/android/web), deviceToken
And duplicate tokens for the same device are updated (not duplicated)
```

## AC-3: Device Deregistration
```gherkin
Given a registered device
When DELETE /api/devices/{deviceId} is called
Then the device registration is removed
And no further notifications are sent to that device
```

## AC-4: Send Notification to User
```gherkin
Given a user has registered devices
When a notification is sent to that user
Then all registered devices for that user receive the push notification
And the notification includes: title, body, optional data payload
```

## AC-5: Send Notification to Tenant
```gherkin
Given multiple users in a tenant have registered devices
When a tenant-wide notification is sent via POST /api/admin/notifications
Then all devices for all users in that tenant receive the notification
```

## AC-6: Device Token Validation <!-- audit-added -->
```gherkin
Given a device registration request
When the device token is validated
Then tokens longer than 500 characters are rejected
And empty/blank tokens are rejected
And platform must be one of: ios, android, web
```

## AC-7: Admin Notification Rate Limiting <!-- audit-added -->
```gherkin
Given an admin sends tenant-wide notifications
When more than 10 notifications are sent within 1 hour
Then subsequent sends are rejected with 429
```

## AC-8: Stale Token Cleanup
```gherkin
Given a push provider returns "invalid token" for a device
When the error is handled
Then the stale device registration is automatically removed
And subsequent notifications skip that device
```
