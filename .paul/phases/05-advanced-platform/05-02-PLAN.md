<objective>
## Goal
Create a CLI tool for the Kelta platform that enables developers to perform CRUD operations, manage collections, and interact with the API from the command line.

## Purpose
A CLI tool is essential for developer workflows — scripting, CI/CD pipelines, data migration, and administrative operations. It provides programmatic access to the platform without a browser.

## Design Principles
- **Standards-based:** Uses the existing JSON:API endpoints
- **Built on existing SDK:** Reuses KeltaClient from kelta-web/packages/sdk
- **TypeScript/Node.js:** Consistent with the frontend stack

## Output
- `@kelta/cli` package in kelta-web monorepo
- Commands: auth login, collections list/describe, records list/get/create/update/delete
- Configuration file (~/.keltarc) for API URL and token storage
</objective>

- KeltaClient: REST client for JSON:API collection CRUD

<acceptance_criteria>

## AC-1: CLI Authentication
```gherkin
Given a user runs `kelta auth login --url https://api.example.com --tenant acme`
When they provide their API token
Then the credentials are saved to ~/.keltarc
And subsequent commands use the saved credentials
```

## AC-2: List Collections
```gherkin
Given an authenticated CLI session
When the user runs `kelta collections list`
Then all accessible collections are displayed in a table format
```

## AC-3: Record CRUD
```gherkin
Given an authenticated CLI session
When the user runs:
  - `kelta records list contacts` → lists records
  - `kelta records get contacts <id>` → shows single record
  - `kelta records create contacts --data '{"name":"John"}'` → creates
  - `kelta records update contacts <id> --data '{"name":"Jane"}'` → updates
  - `kelta records delete contacts <id>` → deletes
Then each operation executes against the API and displays the result
```

## AC-4: JSON Output Mode
```gherkin
Given any CLI command
When the user adds --json flag
Then the output is raw JSON (for piping to jq, scripts, etc.)
```

## AC-5: Error Handling
```gherkin
Given an API error (401, 404, 422, 500)
When the CLI receives the error
Then a clear error message is displayed with status code and detail
And the exit code is non-zero
```

</acceptance_criteria>
